数据名称：Web Server Access Log
数据下载链接：https://zenodo.org/records/12635177

数据打标签方法：
a. 通过编译一个全面的正则表达式模式列表来识别已知机器人的UserAgent字符串，这些模式覆盖了搜索引擎爬虫（如Googlebot、Bingbot）、社交媒体机器人（如FacebookExternalHit）、安全扫描器（如Nmap、Acunetix）、爬虫框架（如Scrapy、Selenium）以及其他常见自动化工具；实现上，使用Python的re模块将每个模式编译为不区分大小写的正则表达式对象，确保能高效匹配混合大小写的UserAgent字符串，从而为后续检测提供基础依据，其技术方案依赖于公开的机器人列表和常见工具签名，确保高召回率已知机器人。
b. 定义了三类可疑行为模式：可疑文件扩展名（如.php、.bak）、漏洞探测路径（如/.env、/.git）和可疑参数（如cmd=、union=），每个类别包含多个正则表达式模式；实现上，将这些模式编译为不区分大小写的正则表达式，并组织成字典结构，便于后续检查URI时快速匹配，其技术方案基于常见攻击向量和网络安全最佳实践，旨在捕捉恶意扫描或异常请求行为，从而增强对未知机器人的检测能力。
c. 通过分析UserAgent字符串的多个特征来识别可疑指标，包括检查长度异常（过短或过长）、缺失浏览器标识（如Mozilla、Chrome）、可疑版本格式（如超长数字序列）和随机字符串模式；实现上，使用字符串操作和正则表达式来提取特征，并返回一个包含布尔标志和详细指标的字典，技术方案基于统计学观察——人类UserAgent通常包含标准浏览器标识且长度合理，而机器人可能使用伪装或随机字符串，从而为行为分析提供补充证据。
d. 分析会话中的行为模式，包括请求频率（计算请求间隔的平均值和标准差以检测过于规律或极快的请求）、URI模式（计算重复请求比例）、状态码分布（计算4xx/5xx错误率）和请求方法（检查非标准方法如PUT、DELETE）；实现上，使用pandas和numpy进行时间序列分析和统计计算，并返回一个包含可疑行为列表和指标的字典，其技术方案依据机器人往往表现出自动化、高频率或错误率高的特征，而人类行为则更随机和多样。
e. 使用预定义的可疑模式检查会话中的每个URI，统计匹配可疑扩展名、漏洞探测路径或参数的请求数量，并计算可疑请求比率；实现上，遍历会话DataFrame的每一行，应用正则表达式匹配，并聚合结果，其技术方案基于威胁情报和常见攻击模式，通过模式匹配快速识别潜在恶意活动，从而在UserAgent检测基础上增加一层防御。
f. 基于IP地址分组请求，然后按时间戳排序，并使用超时阈值（30分钟）将请求分割成会话；实现上，使用pandas的groupby和排序功能，并利用safe_datetime_parse函数安全解析时间戳，处理解析失败的情况，其技术方案遵循Web日志分析的标准实践，确保会话边界合理，以模拟真实用户会话的连续性，为后续标签决策提供上下文。
g.采用五阶段检测流程：首先检查UserAgent是否匹配已知机器人模式，然后分析行为模式（如有多个可疑行为则标记为Bot），接着检查可疑请求比率（超过阈值则标记为Bot），再检查robots.txt请求（结合其他特征），最后进行防误判验证（如会话短且无强信号则改为Human）；实现上，使用条件语句链式检查，逐步细化标签，其技术方案基于防御深度原则，通过多维度证据减少误报和漏报，确保标注的准确性。
h. 在标记为Bot后，进行最终验证：检查会话长度、错误率和UserAgent特征，如果会话短（≤3请求）、错误率低（<10%）且无强可疑特征，则推翻初始标签改为Human；实现上，使用简单阈值比较和特征检查，其技术方案旨在处理边缘案例，避免将短暂或正常活动误判为机器人，从而提高标注精度
